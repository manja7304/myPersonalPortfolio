LangGraph Patterns: ReAct, Supervisor & Plan-and-Execute

Manjunath K GApr 202610 min read

LangGraph models agents as state machines — nodes are functions, edges are transitions. This makes complex multi-agent behavior testable, debuggable, and version-controllable compared to ad-hoc while-loops.

ReAct Loop

The workhorse pattern: Reason → Act → Observe → Repeat. Best for tool-heavy tasks with clear stopping conditions.

graph: START → agent → should_continue?
         ↑                    ↓
         └── tools ←──────── yes/no → END

Use when: single-domain tasks, bounded tool sets, predictable iteration counts.

Supervisor / Router

A supervisor node reads the task and routes to specialist sub-graphs. Each specialist is its own compiled LangGraph with isolated tools.

Use when: multi-domain platforms (web + API + cloud), different expertise per surface, need centralized policy enforcement at the router.

Plan-and-Execute

Planner generates a step list upfront; executor runs each step sequentially with replanning on failure. Reduces wasted tool calls from reactive loops.

Use when: long-horizon tasks (full pentest engagement), cost-sensitive workflows, auditable step-by-step execution logs required.

Reflection Node

After critical steps, a reflection node evaluates output quality: "Is this finding supported by evidence? Is the report section complete?" Catches errors before they compound.

Choosing a Pattern

  • Simple Q&A with tools → ReAct
  • Multi-surface platform → Supervisor
  • Long sequential workflows → Plan-and-Execute
  • High-stakes outputs → Add Reflection to any pattern
Don't over-graph. A 3-node ReAct loop beats a 20-node graph you can't debug. Add complexity only when simpler patterns fail eval metrics.
